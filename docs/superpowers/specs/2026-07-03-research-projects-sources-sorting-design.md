# Сортировка колонок таблицы источников (/projects/next) — design

**Дата:** 2026-07-03
**Статус:** согласовано (brainstorming), готово к плану реализации
**Область:** новый экран `/projects/next`, таблица источников (svar DataGrid).
**Эталон:** `Research Projects v10.dc.html` (стрелки сортировки в заголовке);
поведение — встроенное svar (подтверждено через svar MCP,
`docs/guides/sorting_data.md`).

## Цель

Кликабельные заголовки колонок таблицы источников: клик — сортировка
asc/desc со стрелкой, Ctrl/Meta+клик — мультисортировка с индексом в
заголовке (встроенное поведение svar, режим согласован — мультисортировка).

## Решение

Встроенная сортировка svar: параметр `sort` в конфигурации колонок
`sourceGridColumns()` (`src/lib/ui/research-projects-source-row.ts`).
Стрелки/индексы рисует сам грид; допкода в компонентах нет.
Отвергнута внешняя сортировка (state на странице + кастомный заголовок) —
дублирование готовой функциональности.

## Конфигурация колонок

| Колонка | `sort` | Причина |
|---|---|---|
| `title` | компаратор `(a, b) => String(a).localeCompare(String(b), "ru", { sensitivity: "base" })` | корректная кириллица без учёта регистра |
| `typeLabel` | `true` | простые ASCII-строки |
| `materialsLabel` | компаратор по числу: парсинг «4 317» → 4317 | значение ячейки — форматированная строка |
| `lastSyncedAt` | компаратор чисел; `null` = «самая старая дата»: при asc (старые сверху) null вверху, при desc (новые сверху) null внизу. «Всегда внизу» на колоночном компараторе svar недостижимо (desc = инверсия компаратора) | значения `number \| null` |
| `statusLabel` | `true` | простые ASCII-строки |
| чекбокс (`SELECT_COLUMN`) | нет | служебная |

Компараторы — именованные экспортируемые функции в
`research-projects-source-row.ts` (`compareRuStrings`, `compareMaterialsLabels`,
`compareNullableTimestamps`) — юнит-тестируются напрямую.

svar-контракт (из svar-доков): column-level `sort` принимает `true` или
компаратор `(a, b) => 1|-1|0`, где `a`/`b` — ЗНАЧЕНИЯ ячеек колонки;
компаратор заменяет встроенную сортировку и обновляет `sortMarks`.

## Известный риск и условный шаг

`SourcesGrid` пересобирает `rows` при каждом изменении `selectedSourceIds`
(каждой строке подмешивается `selected` для `GridSelectCell`). Если при
обновлении `data` svar сбрасывает сортировку (проверяется вживую первым
делом), лечение в скоупе: рефактор `GridSelectCell` на реактивное чтение
выделения из `api.getReactiveState().selectedRows` (по образцу
`GridSelectAllCell`), после чего `selected` из rows убирается и rows зависят
только от `sources` (фильтрация всё ещё пересоздаёт rows — это ожидаемо:
сортировка при смене данных фильтра либо сохранится, либо нет — зафиксировать
фактическое поведение живой проверкой; сброс сортировки при смене НАБОРА
данных допустим, при смене ВЫДЕЛЕНИЯ — нет).

## Ограничение обёртки

`ExtractumDataGrid` (обёртка) может фильтровать/типизировать поля колонок —
проверить, что `sort` доходит до svar `<Grid>` (тип `ExtractumDataGridColumn`
должен допускать `sort?: boolean | ((a: unknown, b: unknown) => number)`;
поправить тип при необходимости).

## Тестирование

- Компараторы — юнит-тесты в `research-projects-source-row.test.ts`
  (кириллица/регистр, «4 317» > «339», null внизу в обоих направлениях).
- `sourceGridColumns()` — юнит: у 5 колонок задан `sort`, у select-колонки нет.
- Живая проверка в Tauri (приложение запущено, MCP-мост 9223): клик по
  каждому заголовку (asc/desc, стрелка), Ctrl+клик — вторичная сортировка с
  индексом, «Материалы» сортируются числом (4 317 > 339), null-даты внизу,
  сортировка ПЕРЕЖИВАЕТ выделение строк чекбоксами; поведение при фильтрации
  зафиксировать.

## Не в скоупе

Сохранение сортировки между проектами/сессиями; дефолтная сортировка при
загрузке; сортировка таблицы ConnectFromLibrary.
