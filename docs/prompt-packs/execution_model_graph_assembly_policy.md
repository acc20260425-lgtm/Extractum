# Execution Model & Graph Assembly Policy

Status: v1 draft.

This document defines how Prompt Pack JSON results are assembled from LLM
outputs and deterministic pipeline steps. It does not add fields to the core
contract. It defines ownership, generation order, graph healing, quarantine
behavior, and pre-contract fragment handling.

---

## 1. Purpose

The Prompt Pack JSON Contract describes a graph:

```text
source_refs -> evidence -> claims -> claim_relations -> outputs / pack_data
```

LLMs are good at semantic interpretation, but unreliable at maintaining global
graph invariants in large JSON documents. Therefore the pipeline separates:

- semantic generation by LLM;
- deterministic graph assembly by code;
- validation, healing, retry, quarantine, and audit.

The canonical JSON result is produced by the pipeline assembler, not by using
raw LLM output as the final artifact.

---

## 2. Ownership Map

| Field class | Owner | Examples |
|---|---|---|
| Semantic text | LLM-authored | `claim_text`, `description`, `rationale`, `summary_text`, `reasoning_summary` |
| Type classification | LLM-authored with validation | `claim_type`, `evidence_role`, `barrier_type`, `risk_type` |
| First-order semantic links | LLM-selected from registry | `claim_candidates[].evidence_refs`, `relation_candidates[].source_claim_id` |
| Object IDs | Pipeline-owned | `result_id`, `claim_id`, `evidence_id`, `relation_id`, `warning_id` |
| Derived traversal refs | Pipeline-owned | `claim.source_refs`, `technology.source_refs`, `Video.claim_refs`, `synthesis.source_refs` |
| Ordering conventions | Pipeline-owned | natural sort for `contradicts` source/target order |
| Metadata consistency | Pipeline-owned | `metadata.contains_*`, required `quality_flags` |
| Provenance and audit links | Pipeline-injected | `provenance.stage`, `provenance.audit_refs`, top-level `audit_refs` |

LLM stages may return temporary aliases or candidate-local IDs. Final contract
IDs are assigned by the pipeline.

---

## 3. Topological Generation Sequence

Graph assembly is bottom-up. Later stages operate only on immutable registries
created by earlier stages.

1. `source_ingestion`
   - Pipeline creates `source_refs` and source snapshots.
   - `source_ref_id`, `material_id`, `snapshot_id`, `canonical_url`, and
     `internal_uri` are fixed before LLM claim generation.

2. `fragment_candidate_mining`
   - Pipeline creates deterministic locator windows over source materials.
   - LLM identifies potentially meaningful observations inside those windows.
   - Output is an internal pre-contract fragment registry, not canonical
     `evidence[]`.

3. `claim_extraction`
   - LLM receives only immutable fragment/evidence candidate registries and
     bounded context windows.
   - LLM may select from allowed IDs, but must not invent new IDs.
   - Pipeline assigns final `claim_id` values.

4. `canonical_evidence_generation`
   - Pipeline converts selected fragment candidates into canonical `evidence[]`
     entries after final `claim_id` values exist.
   - One canonical evidence object belongs to one `claim_id`.

5. `claim_linking`
   - LLM proposes relations using fixed `claim_id` values.
   - Pipeline normalizes symmetric relation order and validates references.

6. `pack_data_generation`
   - LLM creates pack-specific projections over the existing graph.
   - Pipeline rebuilds traversal refs and verifies all references.

7. `final_synthesis`
   - Pipeline assembles `outputs`, `metadata`, `quality_flags`, `warnings`,
     `limitations`, and `audit_refs`.

---

## 4. Isolated Registry Principle

For structural graph generation stages, LLM operates under a closed-world
assumption.

Rules for claim extraction:

- LLM receives only immutable `source_registry`, `fragment_registry`, or
  `evidence_seed_registry` entries selected by the pipeline.
- LLM may use only IDs provided in `allowed_*_ids` arrays.
- New source, evidence, fragment, or claim IDs generated by LLM are invalid.
- If available evidence is insufficient, LLM should return an `unknown` or
  `verification_task` candidate rather than inventing a link.
- LLM output should be narrow: semantic text fields, classifications, and
  arrays of allowed IDs.
- Pipeline assigns final IDs, fills derived refs, injects provenance, and runs
  validation.

---

## 5. Conservative Healing vs Hard Validation

Pipeline healing is allowed only when it is semantically safe.

### Auto-Healing

The pipeline may fix these issues deterministically:

- rebuild all derived `source_refs`, `claim_refs`, `evidence_refs`, and
  `relation_refs` traversal fields;
- deduplicate ID arrays while preserving stable order;
- normalize `source_claim_id` / `target_claim_id` for symmetric
  `relation_type = "contradicts"`;
- synchronize `metadata.contains_*` fields with required `quality_flags`;
- assign final contract IDs from pipeline-owned counters;
- remove duplicate relation objects that are exact structural duplicates.

### Hard Validation

These issues require retry, quarantine, partial result, or fail:

- dangling references to missing `claim_id`, `evidence_id`, `source_ref_id`,
  `relation_id`, or pack-specific object IDs;
- required fields missing from objects that are otherwise retained;
- invalid `locator_data` for fragment evidence;
- invalid `claim_id` format where natural sort is required;
- relation evidence that points to evidence outside the relation participants;
- unparseable JSON that cannot be recovered by the parser.

The pipeline must not silently delete dangling references in the canonical JSON
result. Removing a dangling reference without audit hides loss of traceability.

---

## 6. Audit & Quarantine Protocol

When hard validation fails:

1. The pipeline creates a validation finding with `rule_id`, `severity`,
   `layer`, `object_path`, `message`, and `object_refs`.
2. If the stage is retryable, the pipeline may return a compact error report to
   the LLM for one or more repair attempts.
3. If retries are exhausted, invalid objects may be quarantined outside the
   canonical result.
4. A sanitized projection may be produced for UI, but the canonical result must
   preserve the failure through `warnings`, `quality_flags`, `metadata`, and
   `audit_refs`.
5. If traceability loss is material, `result_status` should become `partial` or
   `error`.

Quarantine is preferred over silent deletion. UI consumers may hide invalid
objects, but audit consumers must be able to see that graph assembly failed or
was degraded.

### 6.1 Audit Event Granularity

Graph assembly should emit audit events at operation-batch granularity, not
necessarily one audit record per individual field edit.

Recommended audit granularity:

- one audit event per pipeline stage and operation class;
- affected objects listed in `object_refs`;
- `summary` contains compact counts and operation description;
- detailed diffs live in the external audit store referenced by `audit_uri`,
  not inside the canonical JSON result.

Because `audit_refs.event_type` follows the core enum + `custom_event_type`
pattern, v1 uses `event_type = "custom"` for assembly-specific operations.

Example healing audit ref:

```json
{
  "audit_id": "audit_heal_1",
  "audit_uri": "extractum://audit/project_audit_log/audit_heal_1",
  "audit_store": "project_audit_log",
  "event_type": "custom",
  "custom_event_type": "healing",
  "stage": "final_synthesis",
  "timestamp": "2026-06-12T10:00:00Z",
  "summary": "Rebuilt 3 derived traversal fields in claim.source_refs and removed duplicate refs from 2 arrays.",
  "object_refs": {
    "claim_refs": ["claim_5", "claim_8", "claim_12"],
    "evidence_refs": [],
    "relation_refs": [],
    "source_refs": [],
    "unknown_refs": [],
    "verification_task_refs": [],
    "warning_refs": [],
    "limitation_refs": []
  }
}
```

### 6.2 Quarantine Artifact

The quarantine artifact is outside the canonical result. It may be a file,
database row, object-store blob, or audit-store payload. The canonical result
records only a pointer through `audit_uri`, plus warnings/quality flags that
make the degradation visible.

Recommended external quarantine artifact shape:

```json

  "result_id": "result_001",
```

## 7. Pre-Contract Fragment Registry

Canonical `evidence[]` cannot be generated directly from raw material because
each evidence object must belong to one `claim_id`, and claims do not exist at
the earliest extraction stage.

The pipeline therefore uses an internal pre-contract fragment registry.

### 7.1 Observation Mining

1. Deterministic Locator Backbone
   - Code slices raw material into reproducible windows:
     timestamps for video/audio, `text_range` for articles, comment/reply
     locators for discussion sources, and media regions for images.
   - LLM does not invent `locator_data`.

2. Observation Candidate Extraction
   - LLM processes bounded windows and identifies potentially meaningful
     observations.
   - Candidate output contains a minimal candidate record and may include
     optional fields such as `verbatim_quote`, `candidate_type`, and `salience`.
   - Candidate output is not canonical `evidence[]`.

3. Pipeline Normalization
   - Code deduplicates candidates, keeps locator provenance, trims overly broad
     spans, and builds an immutable `fragment_registry`.

### 7.2 Fragment Candidate Minimum

Fragment candidates are internal pipeline objects, but they still need a stable
minimum shape so later stages can use them safely.

Required fields:

- `candidate_id`;
- `source_ref_id`;
- `fragment_type`;
- `locator_data`;
- at least one text layer: `fragment_text` or `observation_summary`.

Recommended optional fields:

- `context_text`;
- `verbatim_quote`;
- `candidate_type`;
- `salience`;
- `parent_candidate_id`;
- `child_candidate_ids`;
- `language`;
- `snapshot_id`;
- `created_by_stage`.

Minimal example:

```json
{
  "candidate_id": "fragcand_12",
  "source_ref_id": "source_ref_1",
  "fragment_type": "video_timestamp_range",
  "locator_data": {
    "schema_version": "1.0",
    "timestamp_start": 120,
    "timestamp_end": 128
  },
  "fragment_text": "We are moving from demos to real internal pilots.",
  "observation_summary": "Speaker describes a shift from demos to pilots.",
  "candidate_type": "trend_signal",
  "salience": "high"
}
```

### 7.3 Canonical Assembly

During claim extraction, LLM receives the immutable fragment registry and
selects candidate IDs. After claims are validated and final `claim_id` values
are assigned, the pipeline creates canonical `evidence[]`.

If one fragment candidate supports two different claims, the pipeline creates
two separate evidence objects with:

- unique `evidence_id`;
- different `claim_id`;
- identical or equivalent `source_ref_id`, `locator_data`, and `fragment_text`;
- independent `evidence_role`, `confidence`, and provenance as needed.

This preserves the one-claim-per-evidence invariant.

## 8. Semantic Merge, Not Geometric Merge

Overlapping `fragment_candidates` are allowed in the internal registry.
The pipeline must not merge candidates solely because their `locator_data`
overlaps.

Merge is allowed only when candidates represent the same semantic observation:

- same `source_ref_id`;
- compatible `fragment_type`;
- overlapping or near-adjacent locators;
- compatible `candidate_type`;
- semantic equivalence in `observation_summary` and normalized fragment text;
- no loss of claim-level meaning.

When overlap provides useful surrounding context, the broader span should be
retained as a hierarchical parent `context_window`. During canonical assembly,
the narrow candidate populates `evidence.fragment_text` and `locator_data`,
while the broader parent span may map to `evidence.context_text`.

Canonical evidence must remain as narrow as possible to precisely support its
bound claim.

---

## 9. Fragment Deduplication Policy

### 9.1 Golden Rule

False merge is worse than false split.

Excessive splitting of fragment candidates is safe and can be compensated during
claim assembly. Incorrectly merging different semantic observations destroys
evidence granularity, pollutes claim extraction, breaks UI highlighting, and
weakens auditability.

### 9.2 Conservative Deduplication Pipeline

Merge checks run only inside the pre-contract fragment registry.

```text
Fragment candidates
  -> deterministic guards
  -> embedding similarity filter
  -> optional LLM-as-a-judge in deep mode
  -> merge or keep separate
```

#### Step 1: Deterministic Guards

Merge is forbidden unless all hard guardrails pass:

- identical `source_ref_id`;
- identical or compatible `fragment_type`;
- locator overlap or small gap;
- compatible `candidate_type`;
- close `salience`;
- same language and same transcript/snippet snapshot where applicable.

If deterministic guards fail, embedding or LLM judge must not override them.

#### Step 2: Embedding Similarity Filter

Embeddings are used as a candidate filter, not as the sole authority.

Inputs:

- `observation_summary`;
- normalized `fragment_text`;
- optional short context.

Default baseline thresholds:

| Similarity | Decision |
|---|---|
| `< 0.82` | Do not merge |
| `0.82-0.92` | Ambiguous; keep separate in v1 baseline |
| `> 0.92` | Candidate for automatic merge if guardrails pass |

#### Step 3: Optional LLM-as-a-Judge

LLM-as-a-judge is optional and intended for deep or high-cost modes only. It may
operate on ambiguous cases that passed deterministic guards.

The judge receives two isolated candidate cards and returns strict JSON:

```json
{
  "same_observation": true,
  "merge_safe": true,
  "reason": "Both candidates describe the same transition from demos to internal pilots.",
  "preferred_locator": "candidate_a",
  "context_candidate_id": "candidate_b"
}
```

Any result other than `same_observation = true` and `merge_safe = true` keeps
candidates separate.

### 9.3 Final Deduplication Matrix

| Candidate state | Pipeline decision | Contract result |
|---|---|---|
| Exact / near-exact duplicate | Automatic merge | One canonical evidence object per bound claim |
| Same observation, different wording | Merge only above safe threshold and guardrails | One canonical evidence object per bound claim |
| Same span, different observation | Do not merge | Separate evidence objects may share locator |
| Different span, same broader theme | Do not merge | Separate evidence; relation/synthesis may connect claims later |
| Uncertain | Do not merge | Candidates remain separate |

---

## 10. Profile-Based Deduplication Thresholds

Default cosine-similarity thresholds apply only when no source-specific profile
exists. Short, context-dependent, mirrored, or highly structured sources require
stricter profiles.

### Cross-Source Isolation

The pipeline must never merge fragment candidates from different `source_ref_id`
values.

Cross-source duplicates, such as an RSS entry and its web mirror, are handled at
material level through `material_id` or during final synthesis. They are not
fragment merges.

### v1 Baseline Profiles

| Source / locator profile | Merge mode | Guardrails |
|---|---|---|
| `youtube_video` / `video_timestamp_range` | Default, allow adjacent merge | Useful for ASR errors and speaker self-rephrasing; allowed only for small temporal gaps inside the same segment or speaker context |
| `telegram_post` / `post` | Strict | Auto-merge only for near-exact text, same `parent_context`, and narrow time window |
| `comment` / `thread_reply` | Very strict, prefer split | Merge forbidden without matching lineage through `comment_id` / `reply_id` and parent context |
| `rss_entry` + `web_page` mirror | Isolation, no cross-source merge | Deduplicate at material level, not fragment level |
| `document_section` / `text_range` | Section-bounded | Auto-merge only inside the same `section_index` / `page_number` and near-exact text |
| Code / technical snippets | Exact-text only | Merge by normalized exact text, not by `observation_summary` embedding |

These profiles are v1 defaults. Implementations may tighten thresholds for
specific projects, but must not weaken cross-source isolation.

## 11. Payload Size and Pruning

Canonical JSON should remain pointer-heavy and compact. Large audit payloads,
raw invalid objects, detailed diffs, model traces, and quarantine dumps belong
in external stores referenced by `audit_uri`, not inline in the canonical
result.

Production pruning rules:

- keep top-level `audit_refs` lightweight;
- keep `audit_refs[].summary` short and count-oriented;
- store full healing diffs in the audit store;
- store full quarantined objects in the quarantine artifact;
- keep only stable object references in `object_refs`;
- avoid embedding raw prompts, raw model responses, or full source fragments in
  audit records;
- keep canonical `warnings` and `quality_flags` human-readable but concise.

The canonical result should contain enough information to prove that a healing
or quarantine event happened and where to retrieve details. It should not carry
the full operational log.

## 12. Reference Validator and CI/CD

The contract requires a programmable validator in addition to JSON Schema.
JSON Schema can check local shape, but cannot enforce all graph invariants.

Reference validator responsibilities:

- parse and validate core shape;
- check ID uniqueness and referential integrity;
- validate companion schemas for `type_data` and `locator_data`;
- rebuild or check derived traversal refs;
- enforce cross-object pipeline rules from `validation_rules.md`;
- run pack-specific validation based on `pack_id`;
- emit validation findings in the standard finding format;
- fail CI for `severity = "error"` unless the test explicitly expects an
  invalid fixture.

Pack development should include fixtures for:

- valid minimal result;
- valid rich result;
- dangling reference failure;
- invalid locator failure;
- healing-required result;
- quarantine-required result;
- pack-specific edge cases.

## 13. Pipeline Health Telemetry

Healing is useful, but frequent healing is a quality signal. The implementation
should track aggregate pipeline health metrics outside the canonical result.

Recommended telemetry metrics:

- `healed_object_count`;
- `healed_field_count`;
- `healing_event_count`;
- `quarantine_event_count`;
- `retry_count`;
- `retry_success_count`;
- `validation_error_count`;
- `validation_warning_count`;
- `dangling_reference_count`;
- `derived_ref_rebuild_count`.

In v1 these metrics are not added as ad hoc keys to core `metadata`, because
`metadata` has a fixed schema. They should live in monitoring/telemetry storage
and may be summarized through an audit event:

```json
{
  "audit_id": "audit_health_1",
  "audit_uri": "extractum://audit/project_audit_log/audit_health_1",
  "audit_store": "project_audit_log",
  "event_type": "custom",
  "custom_event_type": "pipeline_health",
  "stage": "final_synthesis",
  "timestamp": "2026-06-12T10:05:00Z",
  "summary": "Graph assembly healed 4 fields, quarantined 1 object, and completed after 1 retry.",
  "object_refs": {
    "claim_refs": [],
    "evidence_refs": [],
    "relation_refs": [],
    "source_refs": [],
    "unknown_refs": [],
    "verification_task_refs": [],
    "warning_refs": [],
    "limitation_refs": []
  }
}
```

If a future schema version adds `metadata.pipeline_health`, it should be a
versioned object with stable counters, not a free-form map.

## 14. Final Graph Validation Pass

Before emitting a canonical result, the pipeline should run a final graph pass.

Required checks:

- all `*_refs` point to existing objects or approved external result IDs;
- `parent_result_ids` lineage is acyclic when the validator has access to the
  run-level result graph;
- parent_result_ids lineage is acyclic; a result cannot be its own direct or
  transitive parent;
- `evidence.contributing_evidence_refs` is acyclic;
- `claim_relations` graph is acyclic for relation types that require DAG
  semantics, such as `qualifies` and `supersedes`;
- symmetric `contradicts` relations are normalized and deduplicated;
- derived traversal fields match pipeline recomputation;
- pack-specific graph rules pass.

The final graph pass should run after healing and before audit finalization, so
audit can record both the normalization work and the final validation result.

---

## 15. Implementation Notes

- The pre-contract fragment registry is an internal pipeline artifact. It is not
  part of the canonical JSON result unless a future schema explicitly adds it.
- Graph assembly should emit audit events for normalization, healing, retry,
  quarantine, and final validation.
- Validation rules remain in `validation_rules.md`; this document explains how
  the pipeline should produce data that can satisfy them.
- Pack-specific execution stages may refine this flow, but should not make LLM
  responsible for derived traversal fields or global graph invariants.
