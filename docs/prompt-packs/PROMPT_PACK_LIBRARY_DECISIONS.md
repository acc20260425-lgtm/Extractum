# Prompt Pack Library Decisions

Дата фиксации: 2026-06-06.

Документ фиксирует принятые решения по содержательной библиотеке prompt packs для `Research Control Deck`.
Это не UI/UX-спецификация, а рабочая рамка для разработки самих промптов, JSON-схем, правил доказательности и результатов LLM-анализа.

## 1. Стартовая библиотека prompt packs

Из стартовой библиотеки убран pack `Market research`.

Текущий стартовый набор:

- `Problem / Solution Validation`;
- `Technology Watch`;
- `Reputation / Media Monitoring`;
- `OSINT / Source Verification`;
- `YouTube Video Summary`;
- `Telegram Channel / Chat Summary`;
- `Article / Web Page Summary`.

Обоснование:

- `Market research` пока не является приоритетом.
- Нужен больший упор на исследовательские, доказательные и практические сценарии.
- В библиотеке должны быть как общие аналитические packs, так и специализированные source-specific packs.
- Source-specific packs нужны для качественной подготовки материалов, а общие packs - для более сложных аналитических задач поверх разных источников.

## 2. Problem / Solution Validation

Назначение:

```text
Находить проблемы пользователей, предложенные решения, полезные идеи,
подтверждения работоспособности и ограничения применения.
```

Pack работает в гибридном режиме:

- по умолчанию сам находит заметные проблемы, решения и полезные идеи по корпусу;
- если в главном промпте или `{run_goal}` задана конкретная проблема, фокусируется на ней.

### Что считается рабочим решением

Решение считается потенциально рабочим, если есть хотя бы один из признаков:

- автор проблемы подтвердил, что решение помогло;
- несколько участников подтвердили или повторили решение;
- есть четкие шаги, условия применения и объяснение причины.

Обоснование:

- В пользовательских обсуждениях редко встречается лабораторно строгая проверка.
- Для практического анализа достаточно одного сильного признака, но сила доказательства должна отражаться в статусе.
- Такой подход позволяет не терять полезные решения, но отделять подтвержденное от спорного.

### Типы найденных объектов

Pack ищет не только решения проблем, но и полезные идеи.

```text
item_type:
- solution
- useful_idea
```

Полезная идея попадает в результат, если есть признаки практической пользы:

- кто-то применил идею;
- получил результат;
- повторил подход;
- дал четкий рецепт;
- описал условия применения.

Обоснование:

- Не все ценные находки появляются в формате "проблема -> ответ".
- В обсуждениях часто встречаются удачные практики, обходные пути и идеи улучшения без явной жалобы.
- Разделение через `item_type` сохраняет гибкость и не смешивает разные типы результата.

### Результаты

Pack должен выдавать:

- каталог проблем;
- найденные решения;
- полезные идеи;
- рейтинг надежности решений;
- практический отчет;
- доказательства;
- ограничения;
- случаи, где решение не помогло или остается спорным.

### Статусы решений

Используются статусы:

- `Подтверждено`;
- `Вероятно работает`;
- `Спорно`;
- `Не подтверждено`;
- `Опровергнуто`.

Обоснование:

- Статусы отражают не "истину", а силу подтверждения в собранных материалах.
- Нужно различать подтвержденные решения, вероятные решения, спорные советы и явно опровергнутые варианты.

### Источники

В первой версии pack поддерживает:

- `Telegram`;
- `YouTube transcripts/comments`;
- `Forums`;
- `RSS`;
- `Web`.

Отдельный forum/Reddit-like pack не создается. Форумы и Reddit-like обсуждения покрываются этим pack.

Обоснование:

- Основная ценность форумов и Reddit-like источников для этой задачи - цепочки проблем, решений, подтверждений и опровержений.
- Этого достаточно покрыть через специализированный `Problem / Solution Validation`.
- Отдельный форумный pack раздует библиотеку без явной необходимости в первой версии.

### Ограничения и риски

У решений и полезных идей выделяются явные и выводимые из контекста ограничения:

- версия;
- платформа;
- регион;
- цена;
- уровень навыка;
- зависимость от сервиса;
- риск поломки;
- случаи, где решение не подходит;
- условия, при которых оно сработало.

Обоснование:

- Практическая полезность решения зависит от условий применения.
- Если ограничения не выделять, LLM может представить частный workaround как универсальное решение.
- Выводимые из контекста ограничения нужны, потому что авторы обсуждений не всегда явно формулируют условия.

## 3. YouTube Video Summary

Назначение:

```text
Суммаризировать отдельные YouTube-видео и плейлисты,
сохраняя таймкоды, структуру, тезисы, цитаты, инструкции и проверяемые места.
```

Pack поддерживает:

- отдельное видео;
- плейлист.

Комментарии учитываются как отдельный контекст и доказательства, но не смешиваются с содержанием автора видео.

Обоснование:

- Транскрипт отражает позицию автора или участников видео.
- Комментарии отражают реакцию аудитории, уточнения, возражения и полезные дополнения.
- Смешивание этих слоев ухудшает доказательность и может исказить смысл видео.

### Обработка плейлиста

Для плейлиста используется порядок:

1. определить структуру плейлиста как курса или серии;
2. разобрать каждое видео отдельно;
3. собрать общий вывод по плейлисту;
4. показать повторы, развитие темы, пробелы и спорные места.

Обоснование:

- Плейлист часто является последовательностью, а не случайным набором видео.
- Сначала нужно понять логику серии, затем анализировать отдельные видео в этом контексте.

### Обязательные разделы видео

Стандартный результат включает:

- инфо-карта;
- тип контента;
- наличие пошаговых инструкций;
- целевая аудитория;
- summary;
- ключевые тезисы;
- таймкоды;
- выводы;
- цитаты;
- термины;
- практические шаги;
- спорные утверждения;
- что проверить.

Элементы исследовательского слоя добавляются, если они реально есть в материале:

- гипотезы;
- противоречия;
- связи с комментариями;
- доказательность тезисов.

### Инфо-карта

Инфо-карта содержит:

```text
Название видео (гиперссылка)
Автор / название канала
Подписчики
Метрики: длительность, дата, охват
```

Тип контента:

```text
Обучение
Новости
Интервью
Аналитика
```

Наличие пошаговых инструкций:

```text
Да / Нет
```

Также обязательно указывается:

```text
Целевая аудитория: кому и почему это полезно.
```

### Пошаговые инструкции

Если видео содержит пошаговую инструкцию, сохраняются два слоя:

1. Оригинальные фрагменты с таймкодами и цитатами.
2. Нормализованный алгоритм.

Нормализованный алгоритм:

```text
шаг
действие
условие
ожидаемый результат
риск / ограничение
ссылка на таймкод
```

Обоснование:

- Оригинальные фрагменты нужны для доказательности.
- Нормализованный алгоритм нужен для практического применения.
- Два слоя позволяют не терять связь с источником и одновременно получать удобную инструкцию.

## 4. Telegram Channel / Chat Summary

Назначение:

```text
Суммаризировать Telegram-каналы, комментарии к постам и групповые чаты,
не смешивая разные типы Telegram-материалов.
```

Pack универсальный:

- каналы;
- комментарии к постам;
- группы / чаты.

### Основной результат

Pack выдает:

- дайджест;
- аналитическую сводку;
- практические находки.

Материалы каналов, комментариев и чатов разделяются.

Обоснование:

- Telegram часто смешивает новости, мнения, обсуждения и практические советы.
- Канал, комментарии и чат имеют разную природу и разную доказательную ценность.
- Разделение слоев снижает риск неправильных выводов.

### Обязательные разделы

Результат включает:

- главные темы;
- важные посты;
- summary;
- динамика за период;
- цитаты;
- реакции / комментарии;
- спорные тезисы;
- вопросы и ответы;
- полезные идеи;
- участники / роли;
- настроения;
- сигналы риска;
- проблемы и решения;
- что проверить.

Правило приватности и точности:

```text
Участники / роли показываются как типы участников или публичные авторы сообщений,
если это важно для анализа, без лишнего деанона.
```

### Большие чаты

Для больших Telegram-чатов pack сначала группирует сообщения в темы или ветки, а внутри каждой темы выделяет полезные сигналы:

- вопросы;
- ответы;
- решения;
- конфликты;
- новости;
- инсайты;
- повторяющиеся жалобы;
- полезные идеи;
- спорные утверждения.

Обоснование:

- Суммаризация "всего подряд" плохо работает на шумных чатах.
- Группировка по темам сохраняет контекст обсуждений.
- Выделение сигналов помогает не потерять практическую ценность среди шума.

## 5. Article / Web Page Summary

Назначение:

```text
Разбирать RSS-новости, статьи, документацию, блоги, гайды,
changelog/release notes и обычные web-страницы по конкретной ссылке.
```

Pack один общий для RSS и Web.

Обоснование:

- RSS и web-страницы близки по форме: это опубликованные материалы с основным текстом.
- Разделять их на два pack в первой версии не нужно.
- Различия RSS и Web лучше учитывать внутри prompt pack через классификацию материала.

### Классификация материала

Сначала pack определяет тип материала:

```text
content_type:
- news
- article
- blog_post
- documentation
- guide
- changelog
- report
- landing_page
- reference_page
- other
```

### Обязательные разделы

Результат включает:

- инфо-карта: заголовок, ссылка, автор / издание, дата, тип страницы;
- краткое summary;
- кому полезно и почему;
- ключевые факты;
- ключевые тезисы;
- цитаты / фрагменты;
- что нового или важного;
- спорные утверждения;
- что проверить;
- ограничения материала;
- source refs;
- для RSS: первичность новости, дубли, дата публикации;
- для Web: snapshot URL, дата сохранения, видимый основной текст.

Если это гайд или документация, дополнительно:

- пошаговая инструкция;
- условия применения;
- требования;
- риски;
- ожидаемый результат;
- признаки, что инструкция устарела.

Комментарии к web-странице берутся только если они явно есть на той же странице и пользователь включил их сбор.
Pack не обходит весь сайт.

Обоснование:

- RSS/Web слишком разные по жанрам, поэтому нужна первичная классификация.
- Документация и гайды требуют практической структуры.
- Новости и статьи требуют отделения фактов, тезисов и спорных утверждений.
- Snapshot нужен для воспроизводимости доказательств.

## 6. Technology Watch

Подробная pack-specific schema зафиксирована в:

```text
docs/prompt-packs/technology_watch_pack_spec.md
```

Decision log по pack-specific решениям:

```text
docs/prompt-packs/TECHNOLOGY_WATCH_PACK_DECISIONS.md
```

Статус: v1 baseline для первого pack согласован. Оставшиеся OQ-TW-02 и OQ-TW-03
не блокируют реализацию.

Назначение:

```text
Находить новые технологии, тренды, инструменты и сигналы развития,
оценивать зрелость, ограничения и сравнивать технологии, продукты,
подходы и компоненты.
```

Pack делает:

- поиск новых технологий, трендов, инструментов и сигналов развития;
- оценку зрелости технологий;
- оценку практической применимости;
- выявление ограничений и барьеров;
- сравнение технологий, продуктов, подходов и компонентов.

### Обязательные результаты

Результат включает:

- тренды;
- технологии;
- инструменты;
- сигналы развития;
- ограничения;
- оценка зрелости;
- практическая применимость;
- барьеры;
- что проверить;
- сравнение альтернатив;
- карта игроков / проектов;
- риски внедрения;
- доказательства.

Правило:

```text
Подтвержденные факты, прогнозы и слабые сигналы должны быть разделены.
```

Обоснование:

- Технологический мониторинг без оценки зрелости превращается в список новостей.
- Для аналитика важно понимать не только "что появилось", но и "можно ли это использовать".
- Разделение фактов, прогнозов и слабых сигналов снижает риск переоценки хайпа.

### Шкала зрелости

Используется практическая шкала в `technology_watch_pack_spec.md`:

```text
experiment
pilot
limited_production
production
deprecated
unknown
```

Обоснование:

- Эта шкала понятнее для аналитика, чем TRL.
- Она сразу отвечает на практический вопрос: что с технологией можно делать сейчас.

### Структура `pack_data`

`technology_watch` использует technology-centric модель:

```json
{
  "technology_watch": {
    "technologies": []
  }
}
```

Каждая технология собирает свои `maturity`, `signals`, `tools`,
`adoption_barriers`, `risks`, `recommendations` и traversal-ссылки
`claim_refs`, `evidence_refs`, `source_refs`.

Обоснование:

- Аналитик принимает решение по конкретной технологии, а не по разрозненным
  спискам сигналов и барьеров.
- Technology-centric структура сохраняет связь между maturity, tools,
  barriers, risks и recommendations внутри одного объекта.

## 7. Reputation / Media Monitoring

Назначение:

```text
Отслеживать тональность, репутационные риски, претензии,
спорные темы, конфликты, источники негатива, лидеров мнений,
повторяющиеся нарративы и динамику отношения во времени.
```

Pack делает:

- оценку тональности;
- поиск репутационных рисков;
- выявление претензий;
- поиск спорных тем и конфликтов;
- определение источников негатива;
- выделение лидеров мнений;
- выявление повторяющихся нарративов;
- анализ изменения отношения во времени.

### Объект мониторинга

Pack универсален по объекту мониторинга.

Основное поле:

```text
reputation_target
```

`reputation_target` может быть:

- компанией;
- брендом;
- продуктом;
- персоной;
- проектом;
- темой;
- рынком;
- списком конкурентов.

Дополнительные входные поля:

```text
target_aliases
competitors
period
geo
language
source_scope
```

Обоснование:

- Один и тот же механизм мониторинга нужен и для бренда, и для продукта, и для темы.
- `target_aliases` нужны, чтобы находить сокращения, старые названия, транслитерации и разговорные формы.
- `competitors` нужны для явного сравнительного слоя, если пользователь заранее знает рамку сравнения.

### Конкуренты и альтернативы

Используется гибридный подход:

- явно указанных конкурентов pack сравнивает обязательно;
- автоматически найденные альтернативы показывает как `emerging_comparisons`;
- неожиданные сравнения не смешиваются с заданной рамкой анализа.

Пример:

```text
competitors:
- Product A
- Product B

emerging_comparisons:
- open-source alternative mentioned in forums
- substitute service mentioned in Telegram
```

Обоснование:

- Пользователь должен контролировать обязательных конкурентов.
- Корпус может содержать неожиданные альтернативы, которые важны для репутационной картины.
- Автоматически найденные сравнения должны быть видны, но не должны подменять заданную пользователем рамку.

### Поддерживаемые источники

Pack поддерживает все источники первой версии:

- `Telegram`;
- `YouTube transcripts/comments`;
- `Forums`;
- `RSS`;
- `Web`.

Правило обработки:

```text
Авторский материал, новостной материал и реакция аудитории не смешиваются.
```

Примеры слоев:

- новость из RSS или web-статья;
- авторская позиция из YouTube transcript;
- реакция аудитории из YouTube comments;
- обсуждение в Telegram comments или чате;
- форумная ветка с проблемами и ответами.

Обоснование:

- Репутационный сигнал имеет разную силу в зависимости от источника и контекста.
- Новость, личный отзыв, массовая жалоба и комментарий под видео не должны учитываться одинаково.
- Разделение слоев помогает объяснять, откуда именно появился риск.

### Типы репутационных сигналов

Pack выделяет:

- жалобы;
- претензии;
- негативный опыт;
- похвалу;
- доверие;
- рекомендации;
- позитивный опыт;
- конфликты;
- скандалы;
- обвинения;
- спорные темы;
- нарративы;
- мемы;
- повторяющиеся формулировки;
- лидеров мнений;
- слухи и непроверенные утверждения;
- сравнения с конкурентами или альтернативами;
- изменение отношения во времени.

Рекомендуемые домены причин:

```text
quality
price
support
availability
safety
privacy
ethics
legal
performance
reliability
usability
communication
trust
other
```

Обоснование:

- Репутация складывается не только из негатива.
- Позитивные рекомендации, доверие и устойчивые нарративы часто важнее разовых жалоб.
- Причины сигнала нужны, чтобы объяснять не только "что говорят", но и "почему это важно".

### Обязательные результаты

Результат pack включает:

- краткое репутационное summary;
- общий тон по корпусу;
- карта ключевых репутационных сигналов;
- кластеры жалоб и претензий;
- позитивные драйверы;
- спорные темы и конфликты;
- нарративы и повторяющиеся формулировки;
- лидеры мнений или типы участников, если они важны;
- сравнение с явно указанными конкурентами;
- `emerging_comparisons`;
- динамика по времени;
- различия по типам источников;
- risk register;
- evidence refs;
- что проверить дальше;
- ограничения анализа.

### Risk register

Каждый значимый риск хранится как отдельный объект.

Минимальная структура:

```text
risk_id
title
risk_level
signal_stage
sentiment
reason_domain
description
evidence_refs
affected_target
affected_competitors
source_types
first_seen
last_seen
trend
confidence
recommended_action
```

`trend`:

```text
growing
stable
declining
unclear
```

`recommended_action` не должен быть PR-советом от LLM как истина. Это аналитическая рекомендация, которую пользователь проверяет.

### Оценка риска

Риск оценивается по совокупности признаков:

- сила доказательств;
- повторяемость сигнала;
- рост во времени;
- охват или видимость источника;
- доверие к источнику;
- потенциальный ущерб;
- наличие конфликта или обвинений;
- связь с безопасностью, законом, приватностью или этикой;
- влияние на ключевой объект мониторинга.

Правило:

```text
Высокий риск нельзя ставить только из-за эмоционально сильной единичной цитаты.
```

Единичный эмоциональный сигнал может быть важным, но должен получить стадию `Единичный сигнал` и соответствующий confidence.

### Confidence

Для каждого сигнала показывается `confidence`.

Confidence зависит от:

- количества независимых подтверждений;
- качества source refs;
- согласованности разных источников;
- свежести материалов;
- ясности формулировки;
- наличия прямых цитат;
- наличия опровержений или противоречий.

Важно:

```text
confidence не равен истинности сигнала.
```

Это оценка уверенности системы в том, что сигнал действительно присутствует в корпусе и корректно интерпретирован.

### Доказательность

Каждый значимый репутационный сигнал должен иметь evidence refs.

Минимально:

- цитата или фрагмент;
- source ref;
- тип источника;
- дата;
- контекст;
- роль фрагмента: `supports`, `contradicts`, `background`;
- ссылка на исходный материал или snapshot.

Если сигнал заметен, но доказательств мало, он не повышается до сильного риска. Он остается в зоне:

```text
Слабый сигнал / требует проверки
```

### Границы pack

Pack не должен:

- делать юридические выводы как факт;
- считать обвинения доказанными без подтверждения;
- смешивать слухи с проверенными фактами;
- раскрывать лишние персональные данные участников;
- выдавать эмоциональную интенсивность за репутационный риск;
- автоматически предлагать коммуникационную стратегию без доказательной базы.

Pack может:

- выделять признаки кризиса;
- показывать спорные обвинения;
- объяснять, какие материалы подтверждают или ослабляют сигнал;
- предлагать, что проверить дальше;
- показывать, какие источники формируют репутационную картину.

Обоснование:

- Репутационный мониторинг полезен как связка: тональность + причины + кто распространяет + динамика.
- Одна только тональность не объясняет, почему возникает риск.
- Нарративы и повторяемость помогают отличить случайный негатив от устойчивой проблемы.

### Статусы и оценки

Для репутационных сигналов используются три группы оценок.

Тональность:

- `Позитивный`;
- `Нейтральный`;
- `Негативный`;
- `Смешанный`.

Риск:

- `Низкий`;
- `Средний`;
- `Высокий`;
- `Критический`.

Стадия сигнала:

- `Единичный сигнал`;
- `Повторяющийся сигнал`;
- `Разгорается`;
- `Устойчивый нарратив`.

Обоснование:

- Тональность отвечает на вопрос "какой эмоциональный знак".
- Риск отвечает на вопрос "насколько это важно".
- Стадия отвечает на вопрос "это разовый случай или тренд".

## 8. OSINT / Source Verification

Назначение:

```text
Проверять утверждения внутри корпуса проекта, находить сомнительные места,
выявлять противоречия, строить цепочки доказательств и показывать,
что подтверждено, что опровергнуто, что спорно и что требует дальнейшей проверки.
```

Pack не является просто fact checker. Это доказательная система для аналитика:

- проверяет конкретные факты и утверждения;
- находит сомнительные утверждения;
- находит противоречия и слабые места в источниках;
- строит цепочки доказательств: что подтверждает, что опровергает, что неизвестно.

Обоснование:

- В OSINT-задачах важно не только суммаризировать, но и показывать состояние доказательной базы.
- Аналитику нужно видеть не финальный вердикт LLM, а цепочку: claim -> evidence -> contradictions -> unknowns -> verification tasks.
- Pack должен явно отделять проверенное от спорного, неподтвержденного и опровергнутого.

### Типы проверяемых утверждений

Pack проверяет разные типы утверждений, но применяет к ним разные правила доказанности:

- факты;
- даты;
- числа;
- события;
- имена;
- организации;
- ссылки;
- причинно-следственные утверждения;
- оценочные утверждения;
- прогнозы;
- гипотезы.

Правило:

```text
Факты, оценки, причинные связи, прогнозы и гипотезы нельзя проверять одинаково.
```

Примеры:

- дату события можно подтвердить или опровергнуть прямыми источниками;
- причинную связь можно только обосновать с разной степенью уверенности;
- оценку нужно проверять через критерии и контекст;
- прогноз не может быть "подтвержден" как факт, он может быть обоснован или необоснован;
- гипотеза должна оставаться гипотезой, пока нет достаточных доказательств.

### Статусы утверждений

Для проверяемых claims используются статусы:

- `Подтверждено`;
- `Частично подтверждено`;
- `Спорно`;
- `Не подтверждено`;
- `Опровергнуто`.

Обоснование:

- Эти статусы показывают состояние доказательной базы, а не абсолютную истину.
- Статус `Частично подтверждено` нужен для утверждений, где подтверждена часть фактов, но остаются неизвестные или спорные элементы.
- Статус `Спорно` нужен, когда есть сильные подтверждения и сильные опровержения одновременно.

### Надежность источника

Используется гибридная оценка надежности.

Итоговая шкала:

```text
source_reliability:
- Высокая
- Средняя
- Низкая
- Неизвестно
```

Причины оценки:

- первичность;
- независимость;
- репутация;
- дата;
- конкретность;
- наличие доказательств;
- возможная заинтересованность;
- близость к событию;
- полнота контекста.

Обоснование:

- Простая шкала удобна для UI и чтения.
- Без объяснения причин оценка надежности превращается в непрозрачный вердикт.
- Надежность источника должна быть связана с конкретным claim и контекстом, а не быть глобальной навсегда.

### Противоречия

Pack полноценно работает с противоречиями:

- показывает противоречащие источники рядом;
- объясняет, в чем именно противоречие;
- предлагает, что проверить, чтобы разрешить противоречие.

Типы противоречий:

- дата;
- число;
- участник;
- событие;
- причина;
- оценка;
- контекст;
- интерпретация;
- источник происхождения;
- последовательность событий.

Обязательные поля противоречия:

```text
contradiction_id
claim_ids
conflicting_evidence_refs
conflict_type
description
possible_explanations
resolution_status
verification_tasks
confidence
```

`resolution_status`:

```text
unresolved
partially_resolved
resolved
not_resolvable_from_current_corpus
```

Обоснование:

- Для OSINT мало показать, что источники спорят.
- Нужно объяснить, где именно конфликт: число, дата, причина, оценка или контекст.
- Если противоречие нельзя разрешить по текущему корпусу, это должно быть явно видно.

### Итоговый результат

Pack выдает:

- таблицу утверждений со статусами и evidence refs;
- досье по каждому важному утверждению;
- подтверждения;
- опровержения;
- неизвестные;
- надежность источников;
- карту противоречий и слабых мест;
- список "что проверить дальше".

Минимальная структура claim dossier:

```text
claim_id
claim_text
claim_type
status
confidence
short_assessment
supporting_evidence_refs
contradicting_evidence_refs
context_evidence_refs
source_reliability_notes
unknowns
verification_tasks
last_checked_at
```

### Пресеты контроля

OSINT-pack использует общие пресеты контроля:

- `Быстрый`;
- `Стандартный`;
- `Строгий`;
- `Ручной`.

Но интерпретирует их строже, чем остальные packs.

Ориентировочно:

```text
Быстрый
  Быстрый список спорных утверждений и слабых мест.

Стандартный
  Статусы утверждений, evidence refs, основные противоречия.

Строгий
  Полные досье, надежность источников, цепочки доказательств,
  unresolved unknowns и verification tasks.

Ручной
  Аналитик выбирает, какие claims и проверки запускать.
```

Обоснование:

- Отдельную OSINT-шкалу вводить не нужно.
- Общие пресеты уже есть в продукте.
- OSINT-pack должен быть строже внутри той же общей модели контроля.

### Внешняя проверка

В первой версии OSINT-pack работает по текущему корпусу проекта.

Он может предлагать внешние проверки, но сам их не выполняет.

Правило:

```text
Не писать "проверено внешними источниками",
если проверка была только внутри корпуса проекта.
```

Направление развития:

```text
Если позже будут подключены дополнительные источники или инструменты,
pack сможет запускать дополнительный сбор и внешнюю проверку.
```

Обоснование:

- В первой версии важно не создавать ложное ощущение внешней верификации.
- Если LLM видит только корпус проекта, она должна честно обозначить границы проверки.

### Unknowns и verification tasks

Pack хранит и `unknowns`, и `verification_tasks`.

`unknowns`:

- что неизвестно;
- почему это важно;
- к какому утверждению относится;
- какой источник или тип данных нужен.

`verification_tasks`:

- что проверить;
- где проверить;
- какой результат изменит статус утверждения;
- приоритет;
- связанный claim;
- ожидаемый тип доказательства.

Минимальная структура:

```text
unknown_id
claim_id
unknown_description
why_it_matters
needed_source_type

verification_task_id
claim_id
task
where_to_check
expected_evidence_type
status_change_condition
priority
```

Обоснование:

- "Недостаточно данных" само по себе бесполезно.
- Неизвестные должны превращаться в практические задачи проверки.
- Это помогает аналитику продолжить расследование без потери контекста.

### Типы доказательств

Pack различает:

- прямое подтверждение;
- прямое опровержение;
- косвенное подтверждение;
- контекст;
- первичный источник;
- вторичный источник;
- пересказ.

Эти признаки не являются взаимоисключающими.

Пример:

```text
Один evidence item может быть одновременно:
- прямое подтверждение;
- первичный источник.
```

Обоснование:

- Официальный документ, комментарий, скриншот, пересказ и косвенный намек имеют разную доказательную силу.
- Один источник может выполнять несколько ролей относительно claim.

### Риторика и манипулятивность

Pack оценивает риторику и манипулятивность отдельным слоем.

Он выделяет:

- эмоциональные формулировки;
- подмену фактов оценками;
- неконкретные обвинения;
- преувеличения;
- cherry-picking;
- отсутствие контекста;
- признаки координированного нарратива;
- сильные выводы без доказательств.

Это влияет на `source_reliability` только если признаки явные.

Обоснование:

- Эмоциональный стиль сам по себе не делает источник ненадежным.
- Но явная манипуляция, сильные выводы без доказательств и cherry-picking должны снижать доверие к конкретному утверждению.

### Медиа-доказательства

Для медиа-доказательств в первой версии используется осторожный режим.

Правила:

- изображения, скриншоты, видеофрагменты выделяются как отдельный `media_evidence`;
- анализируется только доступный текстовый слой: описание, транскрипт, OCR, подписи, контекст;
- визуальная подлинность не подтверждается;
- pack явно пишет, что медиа требует ручной или специализированной проверки.

Направление развития:

```text
Проверка подлинности медиа отдельными инструментами.
```

Обоснование:

- LLM не должна делать вид, что надежно проверила подлинность изображения или видео.
- Текстовый слой можно использовать как контекст, но не как доказательство визуальной подлинности.

### Readable output

Финальный человекочитаемый результат включает:

- краткое OSINT summary;
- что можно утверждать сейчас;
- что частично подтверждено;
- что спорно;
- что не подтверждено;
- что опровергнуто;
- карта противоречий;
- ключевые evidence chains;
- unknowns;
- verification tasks;
- ограничения проверки.

Правило:

```text
JSON хранит таблицы, досье и связи.
Readable output дает понятный OSINT-отчет для аналитика.
```

Обоснование:

- Системе нужен структурный JSON.
- Человеку нужен ясный отчет, а не сырой лог.

## 9. Общие правила для всех packs

Все packs должны сохранять:

- JSON-результат для пайплайна и UI;
- человекочитаемое summary;
- source refs;
- доказательные фрагменты;
- ограничения и неизвестные;
- спорные места;
- что проверить дальше.

Общие принципы:

- не смешивать разные типы материалов внутри одного вывода;
- отделять факты, выводы, прогнозы, гипотезы и мнения;
- не повышать статус утверждения без достаточных доказательств;
- сохранять связь результата с исходным фрагментом;
- явно показывать, если данных недостаточно;
- использовать run-scoped snapshots для воспроизводимости.

## 11. Второй pack: `youtube_summary`

Подробная pack-specific schema зафиксирована в:

```text
docs/prompt-packs/youtube_summary_pack_spec.md
```

Decision log по pack-specific решениям:

```text
docs/prompt-packs/YOUTUBE_SUMMARY_PACK_DECISIONS.md
```

### Структура `pack_data`

`youtube_summary` использует video-centric модель:

```json
{
  "youtube_summary": {
    "videos": [],
    "synthesis": null
  }
}
```

Решения:

- `videos[]` является основной атомарной моделью single-video и multi-video runs;
- каждый `Video` имеет обязательный `source_ref_id`, указывающий на top-level
  `source_ref` с `source_type = "youtube_video"`;
- `synthesis` является top-level cross-video слоем и равен `null` при одном видео;
- pack использует `segments`, а не `chapters`, потому что смысловые блоки могут быть
  LLM-generated;
- `video.open_questions` описывает вопросы внутри содержания видео, а top-level
  `unknowns` описывает пробелы анализа pipeline-а;
- `notable_quotes` ограничены коротким verification/evidence use и не предназначены
  для перепубликации transcript-а.

### Pack-specific stages

Pack-specific stages v1:

- `youtube_summary/transcript_analysis`;
- `youtube_summary/segment_extraction`;
- `youtube_summary/key_point_extraction`;
- `youtube_summary/quote_extraction`;
- `youtube_summary/synthesis`.

Открытые вопросы v1 non-blocking:

- ~~OQ-YS-01~~ — закрыт через `speaker_id` в `notable_quote`;
- OQ-YS-02 — granularity `synthesis`;
- OQ-YS-03 — playlist-level context.
