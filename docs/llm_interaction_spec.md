# LLM Interaction Spec

Дата: 2026-06-05.

Проект:

```text
G:\Develop\Extractum
```

Цель документа: зафиксировать принятую систему взаимодействия с LLM для продукта `Extractum`, чтобы по ней дальше проектировать UI, архитектуру пайплайна, хранение результатов и audit log.

## 1. Базовая идея

Приложение использует LLM как аналитический движок для обработки источников:

- Telegram;
- YouTube;
- форумы;
- RSS;
- обычные web-страницы.

Основной режим взаимодействия: **гибридный**.

LLM выполняет пайплайн анализа, но аналитик управляет промптами, видит этапы, проверяет спорные места, подтверждает выводы и запускает дополнительные LLM-действия.

## 2. Пресеты контроля

Уровень контроля задается настройкой проекта через пресеты:

- `Быстрый`;
- `Стандартный`;
- `Строгий`;
- `Ручной`.

Детализация пайплайна зависит от пресета.

Ориентировочная логика:

```text
Быстрый
  Упрощенные этапы, меньше проверок, быстрее получить черновой результат.

Стандартный
  Видны основные этапы, есть проверка ключевых тезисов и спорных мест.

Строгий
  Полная цепочка анализа, полная доказательная база, расширенная проверка качества.

Ручной
  Аналитик сам выбирает этапы, проверки и точки вмешательства.
```

## 3. Подготовка исходных материалов

Подготовка данных зависит от типа источника.

### Telegram

- Хранить посты.
- Хранить комментарии, если они собраны.
- Комментарии анализировать как отдельный слой.
- Важно не смешивать исходный пост и реакцию аудитории.

### YouTube

- Хранить транскрипт видео.
- Хранить таймкоды.
- Хранить комментарии.
- Транскрипт и комментарии анализировать отдельно.
- Для доказательств использовать timestamp.

### Форумы

- Анализировать конкретную ветку целиком.
- Ответы в ветке являются основным материалом, а не дополнительным комментарием.
- Важно учитывать структуру обсуждения и ответы участников друг другу.

### RSS

- Анализировать статью или новость.
- Комментарии не анализировать.
- Важны дата публикации, источник, дубли между лентами, первичность новости.

### Web-страницы

- Анализировать только конкретную страницу, указанную ссылкой.
- Не обходить весь сайт.
- Хранить основной текст страницы.
- Комментарии брать только если они явно есть на той же странице и пользователь включил их сбор.

## 4. Основной LLM-пайплайн

Базовая обработка иерархическая:

```text
фрагмент -> материал -> источник -> тип источника -> проектный отчет
```

Пайплайн должен поддерживать такие этапы:

```text
1. Сбор материалов
2. Очистка и нормализация
3. Разбиение на фрагменты
4. Source-level analysis
5. Summaries по типам источников
6. Извлечение тезисов
7. Построение доказательной базы
8. Поиск противоречий
9. Проверка качества
10. Генерация выбранного варианта отчета
```

В текущей первой версии можно показывать не все этапы одинаково подробно. В `Быстром` режиме цепочка может быть упрощенной, в `Строгом` - полной.

## 5. Формат результатов LLM

Каждый этап LLM возвращает:

- структурированный JSON;
- человекочитаемое резюме.

Примерная структура результата:

```json
{
  "summary": "...",
  "key_facts": [],
  "claims": [],
  "quotes": [],
  "risks": [],
  "contradictions": [],
  "confidence": 82,
  "source_refs": []
}
```

JSON нужен для UI, фильтров, доказательной базы, статусов, audit log и экспорта.

Человекочитаемое резюме нужно для аналитика и отчетов.

## 6. Политика доказательности

Каждый тезис и факт обязан иметь доказательную привязку к исходным фрагментам.

Минимальная доказательная привязка:

- цитата или извлеченный фрагмент;
- source ref;
- ссылка на исходный материал;
- timestamp, post id, comment id, thread id или page snapshot ref, если применимо;
- локальная копия или ссылка на сохраненный материал.

Если LLM видит возможный вывод, но доказательств недостаточно, такой вывод не попадает в отчет. Он отправляется в отдельный слой:

```text
Гипотезы / на проверку
```

## 7. Противоречия

Выбран комбинированный подход.

Система должна:

- иметь отдельный этап поиска противоречий после анализа;
- показывать противоречия в доказательной базе;
- учитывать найденные спорные сигналы в проектном чате;
- при необходимости дополнительно проверять противоречия по вопросу аналитика.

Примеры противоречий:

- разные цифры;
- разные даты;
- противоположные оценки;
- источники, подтверждающие и опровергающие один тезис;
- расхождение между словами автора видео и реакцией комментариев;
- расхождение между формальной RSS-новостью и практическим опытом форума.

## 8. Работа с большими объемами данных

Текущая версия:

```text
Анализировать все собранные фрагменты.
```

Система не должна автоматически отбрасывать материалы через relevance filter по умолчанию.

Стоимость контролируется через:

- предварительную оценку;
- рекомендации по оптимизации;
- выбор модели/пресета;
- возможность сократить объем вручную.

## 9. Поиск по корпусу

Текущая версия:

```text
Без отдельного индекса.
```

Система работает по уже построенному аналитическому слою:

- summaries;
- тезисы;
- доказательства;
- противоречия;
- отчет;
- audit log;
- статусы запусков.

Если нужной информации нет в аналитическом слое, чат не должен фантазировать. Он предлагает запустить дополнительный анализ.

Направление развития:

```text
Гибридный поиск: векторный + ключевой.
```

В будущем такой поиск может использоваться для:

- более сильного проектного чата;
- поиска дополнительных доказательств;
- перепроверки тезисов;
- поиска фрагментов по смыслу и точным совпадениям.

## 10. Проектный чат

Поддерживаются два режима чата:

```text
По аналитическому слою
Свободный анализ
```

Режим по умолчанию:

```text
По аналитическому слою
```

В этом режиме LLM отвечает только на основании:

- summaries;
- тезисов;
- доказательств;
- противоречий;
- отчета;
- audit log.

Если данных нет, чат должен предложить запустить дополнительный анализ, а не придумывать ответ.

`Свободный анализ` может рассуждать шире, но обязан явно помечать, что не подтверждено материалами проекта.

## 11. Быстрые LLM-действия

Рядом с тезисом в доказательной базе нужны быстрые действия:

- `Объяснить тезис`;
- `Найти противоречия`;
- `Добавить доказательства`;
- `Перепроверить`.

Каждое быстрое действие сохраняется в audit log.

### Добавить доказательства

В первой версии ищет только среди уже извлеченных цитат и `evidence candidates`.

Не сканирует весь корпус заново.

### Перепроверить

Запускает LLM-задачу проверки тезиса.

Вход:

- формулировка тезиса;
- связанные доказательства;
- спорные сигналы;
- недостающий контекст;
- текущий статус;
- текущий confidence.

Выход:

- обновленный статус;
- обновленный confidence;
- список проблем;
- рекомендация аналитику.

## 12. Роли LLM

Выбран гибрид ролей.

### Слой 1. Source analysts

Роли, специализированные по типу источника:

- `Telegram analyst`;
- `YouTube analyst`;
- `Forum thread analyst`;
- `RSS analyst`;
- `Web page analyst`.

Они учитывают специфику источника и возвращают единый JSON-формат.

### Слой 2. Task specialists

Роли, специализированные по задаче:

- `claim extractor`;
- `evidence linker`;
- `contradiction checker`;
- `report writer`;
- `quality judge`.

Они работают поверх нормализованных результатов source analysts.

## 13. Проверка качества

Качество проверяется по пресету контроля:

```text
Быстрый
  Только JSON schema validation.

Стандартный
  JSON schema validation + self-check LLM.

Строгий
  JSON schema validation + self-check LLM + LLM-as-judge.

Ручной
  Пользователь сам выбирает проверки.
```

Если LLM вернула невалидный JSON:

1. Сначала выполняется repair-запрос.
2. Затем повторная валидация по схеме.
3. Если не помогло:
   - критичный этап получает статус ошибки;
   - некритичный этап может сохранить частичный результат с предупреждением.

## 14. Статусы

Статусы зависят от типа объекта.

### LLM-задачи

- `Ожидает`;
- `Выполняется`;
- `Ошибка`;
- `Готово`;
- `Повтор`.

### Тезисы

- `Черновик`;
- `Проверено`;
- `Спорный`;
- `Отклонен`;
- `Устарел`.

### Отчеты

- `Черновой`;
- `Частично проверено`;
- `Полностью доказательно`;
- `Устарело`.

### Источники и материалы

- `Новые`;
- `Проанализированы`;
- `Изменились`;
- `Ошибка сбора`.

Для важных результатов показывать confidence score:

```text
Проверено · 82/100
Требует проверки · 54/100
```

Важно: confidence не равен абсолютной истине. Это оценка уверенности по данным, доказательствам и проверкам.

## 15. Устаревание результатов

Выбран комбинированный подход:

- метки устаревания на summaries, тезисах, доказательствах и отчетах;
- панель `Что изменилось`;
- diff после пересчета.

Примеры меток:

- `есть новые материалы`;
- `summary устарел`;
- `тезис требует перепроверки`;
- `доказательная база не обновлена`;
- `отчет частично устарел`.

После пересчета показывать diff:

- что изменилось в summary;
- какие тезисы добавлены/удалены/переформулированы;
- какие доказательства добавлены/удалены;
- как изменился confidence;
- какие спорные сигналы появились.

## 16. Обновление при новых материалах

Обновление зависит от источника:

- RSS - инкрементально;
- Telegram - инкрементально;
- YouTube-комментарии - инкрементально;
- YouTube-транскрипт - обычно стабилен, пересбор по необходимости;
- форумная ветка - инкрементально по новым ответам;
- web-страница - только при ручном обновлении snapshot;
- финальный отчет - обновляется только по команде аналитика.

## 17. Модели и провайдеры

Нужна поддержка нескольких OpenAI-compatible провайдеров:

- OpenAI-compatible endpoint;
- OpenRouter;
- Omniroute;
- любые совместимые gateway через `base_url + api_key + model`.

Система должна иметь слой:

```text
LLM Provider Adapter
```

Он нужен, чтобы пайплайн не зависел от конкретного сервиса.

API-ключи и настройки провайдеров хранятся глобально в защищенном хранилище ОС.

Проекты используют уже подключенных провайдеров.

## 18. Недоступность провайдера или модели

Решение изменено в ходе обсуждения.

Текущая политика:

```text
Если выбранный провайдер или модель недоступны, этап останавливается и показывает ошибку.
```

Автоматического fallback нет.

Пользователь сам выбирает другую модель или провайдера и перезапускает этап.

Ошибка записывается в audit log.

## 19. Выбор модели

Выбран гибрид:

- основной выбор через пресеты качества/стоимости;
- расширенные настройки позволяют переопределить модель для отдельных этапов.

Пресеты:

- `Экономно`;
- `Баланс`;
- `Максимум качества`.

В расширенных настройках можно задать модель для:

- source analysts;
- claim extractor;
- evidence linker;
- contradiction checker;
- report writer;
- quality judge;
- project chat.

## 20. Стоимость и preflight

Перед запуском анализа система показывает предварительную оценку:

- источники;
- материалы;
- примерные токены;
- примерная стоимость;
- выбранный пресет контроля;
- выбранные модели/провайдеры;
- примерное время.

Preflight должен быть минимальным.

Основные действия:

- `Запустить`;
- `Оптимизировать`.

Если оценка высокая, система предлагает оптимизации.

Группы оптимизаций:

```text
Объем
  сократить период;
  только новые материалы;
  исключить комментарии;
  ограничить источники;
  взять top-N материалов.

Модель
  перейти на Экономно;
  оставить сильную модель только для сложных этапов;
  использовать дешевую модель для первичной суммаризации.

Пайплайн
  пропустить поиск противоречий;
  не строить полную доказательную базу;
  сделать только черновое резюме.
```

## 21. Запуск анализа

Основной запуск через одну кнопку:

```text
Запустить анализ
```

После клика открывается preflight.

Пользователь подтверждает запуск, затем стартует LLM-пайплайн.

После завершения анализа открывается вкладка:

```text
Отчет
```

На экране отчета показывается краткая карточка результата:

- `Готово`;
- источники/материалы;
- стоимость;
- время;
- проверенные тезисы;
- спорные сигналы;
- устаревшие части;
- CTA:
  - `Открыть доказательную базу`;
  - `Посмотреть ошибки`;
  - `Сгенерировать варианты отчета`.

## 22. Очередь LLM-задач

Нужна техническая очередь задач.

Показывать:

- этап;
- источник или материал;
- роль;
- модель;
- провайдер;
- статус;
- retry;
- токены;
- стоимость;
- ошибку;
- ссылку на audit log;
- входной/выходной JSON при необходимости.

В UI очередь размещается как нижняя или выдвижная панель.

Она может быть свернута в компактную строку:

```text
LLM: 12 задач · 7 готово · 1 ошибка · $4.20
```

## 23. Audit log

Нужен полный audit log.

Хранить:

- prompt;
- prompt version;
- prompt template;
- модель;
- провайдер;
- параметры;
- входные материалы;
- JSON-ответ;
- repair-запросы;
- ошибки;
- стоимость;
- токены;
- время;
- статус;
- связи с источниками;
- связи с результатами.

Показывать две версии карточки:

### Карточка для аналитика

- что делала LLM;
- какие источники использовала;
- что получилось;
- нужна ли проверка;
- почему есть ошибка.

### Debug/audit карточка

- `job_id`;
- `stage`;
- `role`;
- `provider`;
- `model`;
- `prompt_version`;
- `input_refs`;
- `output_refs`;
- `status`;
- `tokens`;
- `cost`;
- `started_at`;
- `finished_at`;
- `error`.

В UI audit log открывается как drawer справа по клику на задачу или результат.

## 24. Промпты

Используются prompt templates по задачам и типам источников.

Примеры шаблонов:

- `Telegram summary`;
- `YouTube transcript summary`;
- `YouTube comments analysis`;
- `Forum thread analysis`;
- `RSS article analysis`;
- `Web page analysis`;
- `Claim extraction`;
- `Evidence building`;
- `Contradiction detection`;
- `Final report`;
- `Project chat`.

## 25. Обычный режим промпта

В обычном режиме пользователь работает через большой редактор промпта с подсказками.

Показывать:

- основное поле `Промпт исследования`;
- подсказки по хорошему промпту;
- переменные:
  - `{period}`;
  - `{source_types}`;
  - `{control_preset}`;
- preview итоговых промптов для разных ролей.

Пользователь не обязан редактировать внутренние prompt templates.

## 26. Экспертный режим промптов

Экспертный режим реализуется как библиотека prompt templates.

Структура:

```text
Prompt Templates

Source analysts
  Telegram analyst
  YouTube analyst
  Forum thread analyst
  RSS analyst
  Web page analyst

Task specialists
  Claim extractor
  Evidence linker
  Contradiction checker
  Report writer
  Quality judge
```

Для выбранного шаблона показывать:

- текущую активную версию;
- model preset;
- input schema;
- output schema;
- prompt editor.

Минимальные действия:

- `Создать версию`;
- `Сравнить версии`;
- `Откатить`;
- `Сделать активной`.

Каждый LLM-запуск привязан к версии prompt template.

## 27. JSON-схемы

JSON-схемы доступны только в экспертном режиме.

Обычный аналитик видит нормальные UI-поля:

- summaries;
- тезисы;
- доказательства;
- статусы;
- отчеты.

Эксперт может открыть JSON schema для каждого prompt template.

## 28. Редактирование результатов

Вручную редактируются только:

- финальный отчет;
- заметки аналитика.

Summaries, тезисы, доказательства и противоречия не редактируются руками, чтобы не ломать доказательную трассировку.

Их можно:

- принять;
- отклонить;
- пометить как спорные;
- перепроверить;
- пересчитать.

## 29. Приватность и отправка данных

Текущая политика:

```text
Данные отправляются в LLM-провайдера как есть, без маскирования.
```

Отдельного предупреждения или consent-dialog перед отправкой данных не делаем.

Информация о выбранном провайдере и модели остается частью обычных настроек и preflight.

## 30. Хранение LLM-результатов

LLM-результаты и audit log хранятся внутри проекта.

Даже если источники глобальные, результаты анализа проектные:

- summaries;
- тезисы;
- доказательства;
- противоречия;
- отчеты;
- чат;
- LLM job history;
- audit log.

## 31. Связь с глобальными источниками

Текущая версия:

```text
LLM-результаты внутри AnalysisRun ссылаются на run-scoped snapshot и, когда возможно, на canonical SourceMaterial.
```

Каждый analysis run обязан замораживать:

- effective period;
- effective source scope;
- material filters;
- prompt snapshot;
- model snapshot;
- preflight decisions;
- corpus snapshot;
- normalized material text snapshot;
- quote/context snapshots for evidence refs;
- source/material metadata needed for trace.

Live global source library остается входом для будущих запусков. Старые runs, evidence refs, generated reports, drafts and exports читаются через сохраненный run snapshot и не должны молча переезжать на обновленные live materials.

Отдельную project-wide сущность `ProjectSnapshot` в первой версии не вводим. Это не отменяет обязательный `AnalysisRun` snapshot; речь только о том, что общий снимок всего проекта вне конкретного run пока не нужен.

Направление развития:

```text
Project-wide snapshot management and snapshot comparison UI.
```

В будущем можно добавить отдельный UI для сравнения снимков проекта, но он должен дополнять `AnalysisRun` snapshots, а не заменять их.

## 32. Варианты финального отчета

Финальный отчет имеет несколько вариантов:

- `Executive`;
- `Подробный`;
- `Evidence-heavy`;
- `NotebookLM pack`.

Пользователь сам выбирает формат отчета и запускает генерацию конкретного варианта.

Не генерировать все варианты автоматически.

Выбор формата отчета происходит через модалку:

```text
Сгенерировать отчет

Формат:
  Executive
  Подробный
  Evidence-heavy
  NotebookLM pack

Правила включения:
  только проверенные тезисы
  включить гипотезы
  включить спорные сигналы
  включить неполные, но важные наблюдения

Примерная стоимость

Сгенерировать
```

На экране отчета варианты показываются вкладками:

- `Executive`;
- `Подробный`;
- `Evidence-heavy`;
- `NotebookLM pack`.

## 33. Доказательная база и отчет

Связь доказательной базы с отчетом зависит от пресета контроля.

```text
Быстрый
  Упрощенная доказательная база, отчет можно собрать быстрее.

Стандартный
  Доказательства для ключевых тезисов, спорные места помечаются.

Строгий
  Полная доказательная база обязательна, каждый тезис с source refs.

Ручной
  Аналитик сам выбирает правила.
```

Отчет должен показывать уровень доказательности:

- `Черновой`;
- `Частично проверено`;
- `Полностью доказательно`.

## 34. Размещение LLM-системы в UI

Выбран встроенный вариант.

Не делать отдельную большую LLM-админку в первой версии.

LLM-инструменты появляются рядом с основным рабочим процессом аналитика.

Отдельные вкладки `Запуски` и `Промпты` допустимы как сфокусированные аналитические поверхности. `Запуски` показывают run history, snapshots, частичные outcomes и audit refs. `Промпты` - это экспертная библиотека prompt packs: активные версии, черновики, архив, source-type coverage, changelog и публикация. Это не является отдельной LLM-админкой, потому что вкладки не вводят управление ролями, провайдерами, глобальными политиками или отдельный админский слой поверх продукта.

### Preflight

Модалка после кнопки `Запустить анализ`.

### LLM queue

Нижняя или выдвижная техническая панель.

### Audit log

Правый drawer по клику на задачу или результат.

### Project chat

Вкладка или режим правого inspector.

### Prompt editor

Остается в inspector.

### Prompt template library

В текущем прототипе экспертная библиотека prompt packs доступна во вкладке `Промпты`. Обычный prompt editor остается в inspector, а библиотека служит экспертным рабочим экраном для версий prompt packs и их готовности к preflight.

### Устаревание

Показывается метками на объектах и через панель `Что изменилось`.

## 35. Минимальный UI-набор первой версии

Для первой версии LLM-взаимодействия достаточно:

- preflight modal;
- LLM queue bottom panel;
- audit drawer;
- project chat side panel;
- prompt editor in inspector;
- `Запуски` для истории runs, snapshots и audit refs;
- `Промпты` для экспертной библиотеки prompt packs;
- report generation modal;
- LLM-действия в доказательной базе.

Расширенные элементы можно добавить позже:

- полноценный экран провайдеров и моделей;
- глубокий editor для prompt template library;
- экран качества/валидации;
- project-wide snapshot management UI;
- гибридный поиск;
- более подробная панель diff.

## 36. Ключевые не-цели первой версии

В первой версии не делать:

- автоматический fallback провайдеров;
- отдельный vector/hybrid search index;
- separate project-wide snapshots outside `AnalysisRun`;
- ручное редактирование summaries/тезисов/доказательств;
- полную LLM-админку как отдельный раздел;
- обязательное предупреждение об отправке данных провайдеру;
- автоматическую генерацию всех вариантов отчета.

## 37. Короткое резюме решений

```text
Режим работы: гибридный.
Контроль: пресеты Быстрый / Стандартный / Строгий / Ручной.
Пайплайн: иерархический.
Роли: source analysts + task specialists.
Формат: JSON + readable summary.
Доказательность: каждый тезис/факт должен иметь source refs.
Слабые выводы: в Гипотезы / на проверку.
Чат: по аналитическому слою по умолчанию.
Поиск: без индекса в первой версии; гибридный поиск как развитие.
Провайдеры: OpenAI-compatible, OpenRouter, Omniroute.
Fallback: нет, этап останавливается с ошибкой.
API keys: глобально в защищенном хранилище ОС.
Audit log: полный.
Промпты: templates по источникам и задачам.
Обычный prompt UI: большой редактор с переменными и preview.
Экспертный prompt UI: template library.
Хранение результатов: внутри проекта.
Связь с источниками: run-scoped snapshots + canonical SourceMaterial refs when available.
Отчеты: пользователь генерирует выбранный формат.
UI: LLM встроена в существующие экраны.
```
